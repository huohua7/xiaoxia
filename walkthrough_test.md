# 端到端测试 Walkthrough

## 虚构申请人
张明，某985电子工程大四，GPA 3.4，目标CMU MSCS / UIUC MCS / UCSD MSCS

---

## 阶段零：诊断

### Coordinator 动作
1. 收到张明上传的简历和目标项目列表
2. 并行启动：自己做诊断 + 调用 program_researcher 分析三个项目

### Program Researcher 产出（以CMU为例）

走 program_researcher.md 的模板，应该产出：
- 基本信息 + 文书要求清单
- 招生偏好
- 典型录取者画像
- 项目独有特色
- 文书写作提示

【发现问题A】program_researcher 需要搜索CMU MSCS的信息。但如果搜索结果中关于"典型录取者画像"的公开数据很少（CMU不公布录取统计），program_researcher会怎么处理？当前prompt说"标注信息不足"，但没有说在信息不足时诊断报告的匹配度评估应该怎么降低置信度。

建议修复：在CLAUDE.md诊断报告模板的匹配度表格中增加一列"评估置信度"，当 program_researcher 报告中标注了"录取者画像信息不足"时，该项目的匹配度评估标注为"低置信度"，提醒用户这个评估可能不准。

### Coordinator 诊断报告

基于简历，大概会产出：
- 背景概要：985 EE，GPA 3.4，1段实验室+1段实习+1个课程项目
- CMU MSCS：匹配度低（GPA偏低，无论文，EE转CS）
- UIUC MCS：匹配度中（MCS对研究要求较低，工程经历有帮助）
- UCSD MSCS：匹配度中（相对友好，但竞争激烈）

【发现问题B】诊断报告中的"建议"列，当前只有"正常推进/文书需重点弥补XX/建议更换"三种。但对于张明这种GPA 3.4申CMU的情况，诚实的建议可能是"录取可能性较低，文书能弥补的空间有限，建议将CMU作为冲刺校同时增加保底校"。当前模板没有"冲刺校/匹配校/保底校"的分层概念。

建议修复：在匹配度表格的"建议"列增加选项：
"作为冲刺校保留（录取可能性低但值得一试）"
"匹配校，正常推进"
"保底校，文书不需要过度投入"

### 用户确认

张明看到诊断报告，可能会：
a) 接受建议，去掉CMU
b) 坚持保留CMU作为冲刺
c) 问"我还能做什么来提高竞争力"（触发背景提升建议）

如果选b，系统尊重用户决定，继续。CLAUDE.md 9.3已经覆盖了这种情况。✓

## 阶段一：素材挖掘

### Interviewer 启动

收到：张明的简历 + 目标列表（"CMU MSCS, UIUC MCS, UCSD MSCS"，无招生偏好详情）

第一步：生成理解文档初始版本。

初始版本大概长这样：

```
## 核心画像摘要
EE转CS的应届生，GPA中等偏下，有一段信号处理实验室经历和一段前端开发实习，
加一个编译原理课程项目。三段经历横跨硬件信号处理、前端开发、编译器，
领域跨度大。初步假说：这个跨度背后可能有一条从EE向CS转变的动机线索，
需要在对话中验证。素材总量偏少，需要挖深每段经历的细节。

## 待验证假说
- 假说1：EE到CS的转变有明确的动机触发点 ← 来源：简历中大三开始选CS课程
- 假说2：实验室经历可能是负面的（导致他不想继续做硬件） ← 来源：推测，无直接证据
```

看起来合理。✓

### 阶段一：破冰

Interviewer 列出三段经历，问情感温度。

张明可能回答：
- 信号处理实验室："还行吧，有点无聊"
- 前端实习："挺喜欢的，节奏快"
- 编译器项目："还行"

热力图：
- 实验室：中温（"无聊"是负向信号，可能藏价值观线索）
- 实习：高温（正向，优先深挖）
- 编译器：低温（暂搁置）

然后 Interviewer 问英文写作样本。张明说"没有什么正式的英文写作，就写过英文邮件"。
→ 理解文档标注"无英文写作样本"。✓

问推荐信。张明说"找实验室导师和实习主管"。导师可能写研究能力，实习主管可能写工程能力。
→ 理解文档记录推荐信覆盖情况。✓

展示热力图给用户确认。
→ 张明可能说"其实编译器那个我也挺喜欢的，做到最后能跑通程序的时候特别爽"
→ 编译器升温到高温。

【发现问题C】Interviewer 展示热力图让用户确认时，用户可能不只是确认，还会补充新信息（如上面编译器的例子）。当前prompt说"让用户确认有没有遗漏"，但没有说如果用户在确认环节给出了新的情感信号，interviewer 应该立即更新热力图。这不是大问题，但值得明确。

### 阶段二：深挖

先挖高温经历：前端实习。

场景还原：
"你在实习的时候，每天大概做什么？"
张明："就写前端代码呗，用React，然后有时候开会"
"写React的时候是自己独立做一个功能还是和别人一起？"
张明："一开始跟着师兄学，后来自己独立做了一个内部工具的前端"
"内部工具是什么工具？"
张明："一个数据看板，让运营的人能看各种指标"

这里 Interviewer 应该注意到"能量变化"——张明提到"自己独立做"和"让运营的人能看"这两个点时，如果语气变生动，说明有素材。✓

角度探测（给2-3个角度让张明选）：
A："这段经历让你发现自己喜欢独立把一个东西从零做出来"
B："这段经历让你发现自己喜欢做能被人直接用到的东西"  
C："这段经历是你确认自己要从硬件转软件的关键时刻"

张明选B："对对对，就是看到运营那边的人真的每天在用我做的看板，特别有成就感"

→ 更新理解文档。✓

然后挖编译器项目和实验室经历，过程类似。

### 阶段三：模式发现

假设三段经历深挖后，发现：
- 实习：喜欢做能被人直接用的东西
- 编译器：程序跑通那一刻的成就感
- 实验室：觉得无聊的原因是"做了很多处理但不知道最后有什么用"

Interviewer 发现模式："你在每段经历中的兴奋点都和'看到自己做的东西产生实际效果'有关。实习时是运营在用你的看板，编译器是代码真的能跑，实验室让你无聊恰恰因为看不到输出的价值。你有没有发现这个共同点？"

张明："嗯好像是的，我确实比较受不了做了半天看不到结果的事"

→ 跨经历模式：可信度"强"（三段经历自然指向同一主题，用户强烈认同）✓

【发现问题D】这个模式发现非常自然。但如果张明的三段经历真的毫无关联呢？当前设计里有"无模式备用策略（成长序列）"在 strategist 端。回看 interviewer.md... 已经写了："如果没有发现有说服力的模式，诚实记录'未发现强模式'，不要为了交差硬编一个。" ✓ 已覆盖。

### 退出

三段经历深挖完成 + 每段有叙事角度 + 有一条强模式 → 条件A满足 → 退出 ✓

- 实习：简历写"负责数据看板前端开发"，理解文档中是"独立完成内部工具前端"，一致。
- 无重大偏差 → 跳过修正 ✓

## 阶段二：策略规划

Strategist 收到：
- 完整理解文档
- 三个项目的分析报告
- 诊断匹配度评估

### 核心叙事主线

基于强模式"追求看到实际效果"，主题句可能是：
"我是一个被'让技术产生可见影响'驱动的人，从信号处理到前端开发到编译器，每一步都在寻找让代码变成可用产品的最短路径。"

### 推荐信协同

推荐信覆盖：导师→研究基础，实习主管→工程能力
PS侧重：个人动机、转方向的心路历程、对"技术产生可见影响"的追求 ✓

### 竞争定位

CMU：GPA低于典型录取者（3.7+），研究经历弱。独特卖点：有实际产品开发经验。
UIUC：GPA勉强够。独特卖点同上。
UCSD：GPA匹配。独特卖点同上。

### 文书分组

三个项目都是CS硕士，但CMU的文书要求可能不同（可能有specific prompts）。
假设CMU和UCSD都是开放式PS，UIUC也是。
→ 可以分为一组，以UIUC为种子（匹配度最高）。

【发现问题E】种子文书基于UIUC的特点写，然后改写给CMU，CMU的文书可能不够"有野心"。冲刺校的文书策略应该更aggressive，匹配校的文书可以更balanced。

建议修复：在 strategist 的分配方案中增加"申请策略基调"维度。

### 面试可行性预判

检查每个模块的面试风险：
- 实习模块：角度是用户自己选的 → 低风险 ✓
- 编译器模块：角度是用户自己选的 → 低风险 ✓  
- 实验室模块：如果被用来做"从EE转CS的动机"叙事，角度可能有微调 → 中风险 ✓

### 策略质量自检

核心叙事能一句话说清 ✓
每个模块自包含 ✓
字数分配合理 需检查
引用可追溯 ✓

## 阶段三：文书撰写

### Writer 写CMU PS

收到：策略简报CMU部分 + 理解文档相关板块 + CMU项目分析 + banned_expressions.md

风格校准：无英文样本 → 使用默认清晰简洁风格 ✓

开头：按策略简报指定的场景（可能是实习时看到运营用数据看板的那个时刻）

【发现问题F】Writer 写项目契合段时需要提到CMU的具体特色。但 program_researcher 的报告中列出的教授/课程/实验室可能有几十个。writer 怎么知道该挑哪个提？这个匹配判断应该在 strategist 阶段完成。

建议修复：在 strategist 的结构大纲中，项目契合段明确指定要提及的1-2个具体特色。

## 阶段四：质量审核

Writer 提交 draft_cmu_v1.md。

三方并行：用户审阅 + Reviewer + Consistency Checker（此时只有一篇，暂不触发）

### Reviewer 审核

走六维度 + AI痕迹双层检查。

【发现问题G】Reviewer 需要读 banned_expressions.md 做词汇层兜底检查。但 coordinator 在调用 reviewer 时需要把这个文件路径传过去。

建议修复：在 CLAUDE.md 阶段四的描述中明确写出"调用 reviewer 时额外传递 banned_expressions.md 的文件路径"。

### 用户审阅

张明看到初稿，说"开头还行，但第二段把我说得太厉害了，我其实就是做了个很简单的看板，不是什么复杂系统"。

这属于事实准确性反馈 → 最高优先级 → 传给 writer 修正 ✓

## 阶段五：面试准备

Interview Coach 生成模拟问题。

对于"从EE转CS"这个叙事线，可能生成：
"你为什么决定从电子工程转到计算机科学？"

【发现问题H】如果张明的真实回答是"因为CS工资高/就业好"，这在面试中是一个很差的回答。Interview Coach 当前只生成问题和回答要点，但没有标注"哪些回答方向是面试中的雷区"。

建议修复：Interview Coach 增加"避免的回答方向"字段。
