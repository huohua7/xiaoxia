# 🦐 小虾 XiaoXia — AI多Agent研究生文书生成系统

> 把留学文书顾问的经验系统化之后，开源给买不起顾问的人。

## 这个项目怎么来的

我们是一个四个人的文书团队，从2019年做到现在，经手过大约四百个研究生申请，美英港新都做。四个人分别覆盖社科（教育学、公共政策、传播学）、工科（CS、EE、ME）、理科（化学、生物、统计）、商科（金融、MBA、管理学），PhD和硕士都带过。

做这行久了会积累出一些直觉。比如听一个学生讲完实习经历，你大概能判断这段素材能不能用、该从什么角度切入。但这种直觉很难传授，我们给新人培训的时候就发现，很多判断说不清楚为什么，就是"感觉这样比较好"。

2025年初我们想做一件事：把这些直觉逼成规则。不是让AI替我们干活，而是看看我们的方法论能不能被形式化地表达出来。花了几个月互相拷问，一个人说"这段经历我觉得值得挖"，另一个人就追问"你凭什么觉得值得挖，你脑子里到底在判断什么"。最后把这些东西整理成了一套多Agent系统的配置文件。

开源的原因很简单。文书咨询本质上是个信息差生意，好顾问贵是因为他有判断力。但一个家庭条件好的学生花几万块就能买到的东西，另一个同样优秀但没有预算的学生只能自己摸索，这中间的差距和两个人本身的能力差距完全不成比例。AI让我们有机会把经验开放出来，至少能让更多人拿到一套及格线以上的指引。

## 做了什么

一套prompt配置文件，定义了7个专业Agent的协作流程。喂给Claude之后，它可以完成从竞争力诊断到面试准备的全流程。支持PS、SOP、Diversity Statement、Research Statement和短问答。

工作流分六步。先做竞争力诊断和目标项目分析。然后通过对话挖掘申请人的素材，产出一份理解文档。基于理解文档做叙事策略设计和跨项目素材分配。之后写文书。写完由审核Agent和一致性检查Agent并行审读，用户也同时看。最后生成面试准备材料。

## 几个设计上的判断

文书的价值主要在写之前。大部分学生以为核心工作是"写"，其实是素材挖掘和策略规划。同样的经历换一个叙事角度，说服力可以差很多。所以系统前三个阶段的权重远高于写作本身。

素材挖掘阶段用了我们说的"脚手架式提问"。你问一个学生"谈谈你的实习经历"，他会卡住。你问"你在实习的时候每天到公司第一件事做什么"，他就能开始说了。整个Interviewer的提问逻辑围绕这个思路设计。

我们发现好的PS通常有一条贯穿全文的隐含主线，来自申请人不同经历之间的关联。系统在素材挖掘的第三阶段专门做这件事。

推荐信和PS经常内容重复，都在讲"研究能力强"，浪费了一个独立的信息通道。系统会收集推荐信信息，策略阶段据此调整PS的侧重。

冲刺校和保底校的文书写法应该有区别。系统会基于诊断结果给每篇文书标注不同的基调。

AI痕迹的问题主要在结构层。禁掉几个套话词只是表面工作，真正暴露的是每段差不多长、每个论点恰好配一个例子、情感从头到尾平滑上升这种机械感。审核Agent做的是词汇和结构双层检测。

```
├── walkthrough_test.md          # 端到端测试记录
├── agents/
│   ├── interviewer.md           # 素材挖掘
│   ├── program_researcher.md    # 项目调研
│   ├── strategist.md            # 策略规划
│   ├── writer.md                # 文书撰写
│   ├── reviewer.md              # 质量审核
│   ├── consistency_checker.md   # 一致性检查
│   └── interview_coach.md       # 面试准备
└── projects/                    # 用户数据（不上传）
```

## 用法

**Claude Code：** 克隆仓库，把 `CLAUDE_CODE_PROMPT.md` 内容放进项目的 CLAUDE.md，开始对话。

**Claude Project：** 把 `CLAUDE.md`、`agents/` 下所有文件、`banned_expressions.md` 上传到 Project Knowledge。Project Instructions 里写一句"你是研究生文书生成系统的协调器，读取Knowledge中的配置文件，按六阶段工作流工作"就行。

**System Prompt：** 空间紧张的话按优先级塞。`CLAUDE.md` 最重要，然后 `interviewer.md`，然后 `writer.md` 加 `banned_expressions.md`。

## 防护

系统会拦截writer编造的细节，要求每个叙事点都能追溯到用户确认过的事实。如果用户声称做了某件事但追问两轮给不出任何具体场景，系统会在理解文档里标注可信度存疑，后续自动降低对这段素材的依赖。素材不够一段完整经历的情况下系统会拒绝继续，给用户几个问题让他们回去想。

## 局限

系统的天花板是模型本身的英文写作能力。英文好的申请人自己写可能更好。真人顾问掌握的行业内部信息（哪个项目今年偏好变了、哪个教授不怎么招人了）AI拿不到。素材挖掘的效果取决于用户愿意投入多少，碰上不配合的用户系统也没办法。

## 迭代

系统经过四轮修改。第一版是基础架构，拿去跑了几个case之后发现一堆衔接问题，改了一轮。然后自己做对抗测试，故意用各种极端场景去破坏，又改了一轮。最后用虚构申请人端到端走了一遍完整流程，`walkthrough_test.md` 里有记录。
