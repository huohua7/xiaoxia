# 🦐 小虾 XiaoXia

一套多Agent的研究生文书系统，喂给Claude就能跑。

## 背景

我开留学文书工作室的，四个人的小团队，2019年到现在。社科、工科、理科、商科四个方向都覆盖，硕士PhD都带，美英港新都做过，大概经手了四百多个学生。

做久了你会发现这行最值钱的环节其实不是写。很多学生（包括一些同行）以为文书顾问的核心能力是英文写作，不是的。核心能力是你听完一个学生讲他的经历之后，能判断这段东西能不能用、从什么角度切、跟其他经历怎么串起来。这个判断力才是客户花几万块买的东西。

举个例子。去年有个学生，本科学化学的，想转CS申美硕。简历上就一段实验室经历加一段跟计算机沾点边的课程项目，看起来素材很薄。但我们做完访谈发现，他在实验室里自己写过一个python脚本来批量处理光谱数据，当时就是嫌手动处理太慢自己学着搞的。这件事他简历上提都没提，因为他觉得那个脚本很简陋不值一说。但这恰恰是最好的素材，因为它同时说明了两件事：他转CS的动机是从实际需求出发的，以及他是那种碰到问题会自己动手解决的人。

这种判断，靠的是经验积累出来的直觉。问题是这种直觉很难传授。我们给新人培训的时候试过，你跟他说"要注意捕捉学生回答中的能量变化"，他听完似懂非懂，做出来的访谈还是像在走流程。

2025年上半年我们尝试做了一件事：看看这些直觉到底能不能被写成明确的规则。四个人花了差不多三个月互相逼问。一个人说"这段经历我直觉觉得能用"，其他人就追着问"你到底凭什么觉得能用，你脑子里在判断什么"。逼到最后确实能拆出来一些东西。比如我们发现，判断一段经历值不值得深挖，最可靠的信号不是这段经历本身有多厉害，而是学生在讲到这段经历的时候有没有突然变得具体——突然开始说人名、说时间、说"当时我记得特别清楚"。一旦出现这种变化，基本上挖下去都能出东西。

我们把这类规则整理出来，写成了一套AI能跑的配置文件。不是那种"帮我写一篇PS"的一次性prompt，是一整套工作流：先做诊断，然后有一个专门做访谈的Agent跟学生对话挖素材，挖完了有另一个Agent做叙事策略，然后才开始写，写完还有审核和一致性检查。每个环节对应一个Agent，每个Agent的prompt里都塞满了我们这些年总结出来的规则。

为什么开源？老实讲，我们这行赚的就是信息差的钱。一个学生家里条件好，花三四万找我们，我帮他把一段很普通的实习经历换个角度讲，文书质量就能上一个台阶。另一个学生可能本科背景更好、经历也更扎实，但他没这个预算，自己写出来的PS开头还在用"Ever since I was a child"。这两个人之间的差距跟他们本身的能力没有任何关系，纯粹是信息差。

这套系统不能完全替代真人顾问。我知道NYU那个新来的ad今年偏好什么风格，我知道CMU有个教授的邮箱发套磁信过去大概率不回，这些东西AI拿不到。但方法论层面的东西，什么时候该追问、推荐信和PS怎么做内容分工、冲刺校和保底校的文书基调应该有什么区别，这些不应该是只有花得起钱的家庭才能接触到的信息。

## 系统概览

7个Agent的协作，走六步：

**第零步**是诊断。系统基于学生的简历和目标项目给一个竞争力评估，告诉你哪个是冲刺校哪个是保底校。同时另一个Agent去查目标项目的招生信息。

**第一步**是素材挖掘，这是整个系统最重要的环节。一个专门的Interviewer Agent跟学生进行对话，按照我们设计的三阶段协议走：先快速扫一遍所有经历做个温度排序，然后对重点经历逐段深挖，最后在所有经历之间找跨经历的隐含关联。我们管最后这步叫"连点成线"，好的PS通常都有一条贯穿全文的暗线，这条线就是从这一步出来的。

这个环节我们做了很多细节设计。比如提问策略用的是我们说的"脚手架式提问"。你问学生"谈谈你的实习"他会卡住，你问"你在实习的时候每天到公司第一件事做什么"他就能说了。再比如有个"细节密度异常检测"的机制：如果一个学生说自己"负责了核心系统的重构"，但你追问两轮他连用什么语言写的都答不上来，系统会自动标记这段经历可信度存疑，后面写文书的时候会降低对这段素材的依赖。

**第二步**是策略。一个Strategist Agent拿到素材之后设计核心叙事、把素材拆成可复用的模块、决定每篇文书用哪些模块、从什么角度切入。如果学生申请多个项目，这步还会做跨项目的素材分配，确保不同项目的文书之间有足够的差异。这里面有个设计点：系统会把推荐信已经覆盖的能力维度和PS错开，避免信息重复。

**第三步**才开始写。Writer Agent按策略简报撰写英文文书。支持PS、SOP、Diversity Statement、Research Statement和短问答。写作风格会参考学生的语言样本做校准，如果学生能提供一段自己写过的英文文本效果最好。

**第四步**是审核。一个Reviewer Agent模拟招生官视角做六维度审读，另一个Consistency Checker检查多篇文书之间有没有事实矛盾。学生自己也同时看初稿给反馈。三方意见合并之后决定要不要改。

AI痕迹检测这块我们花了不少心思。一开始只做了个禁用词清单（类似"ignited my passion""transformative experience"这类AI高频表达），后来发现光禁词不够。真正让一篇文书读起来像AI写的，是结构上的机械感：每段差不多长、每个论点恰好配一个例子、情感从头到尾平滑上升没有一点起伏。真人写的东西不会这么工整。所以审核Agent现在做两层检测，结构层的权重比词汇层高。

**第五步**是面试准备。一个Interview Coach Agent预判招生官读完文书之后会追问什么，给每个问题评风险等级。如果文书里有某段叙事角度是系统帮学生重新提炼过的（跟学生自己原话的差距比较大），会标成高风险，提醒学生面试前重点准备这个点。

## 文件

```
├── CLAUDE.md                    # 协调器配置（工作流、规则、伦理、防护机制全在这里）
├── CLAUDE_CODE_PROMPT.md        # 给Claude Code用的操作手册
├── banned_expressions.md        # 禁用表达清单
├── walkthrough_test.md          # 拿虚构申请人跑的端到端测试
├── agents/
│   ├── interviewer.md           # 素材挖掘（最长的一个，三阶段对话协议都在里面）
│   ├── program_researcher.md    # 项目调研
│   ├── strategist.md            # 策略规划
│   ├── writer.md                # 文书撰写
│   ├── reviewer.md              # 审核
│   ├── consistency_checker.md   # 一致性检查
│   └── interview_coach.md       # 面试准备
└── projects/                    # 用户数据，gitignore了
```

## 用法

**Claude Code**（推荐）：克隆仓库，把 `CLAUDE_CODE_PROMPT.md` 内容放进项目的CLAUDE.md，直接开始对话。

**Claude Project**：把 `CLAUDE.md`、`agents/` 下所有文件、`banned_expressions.md` 传到Project Knowledge里。Project Instructions 写一句"你是研究生文书生成系统的协调器，按Knowledge里的配置文件工作"就行。

**System Prompt**：空间紧张就按优先级塞。`CLAUDE.md` 最重要，然后 `interviewer.md`，然后 `writer.md` 加 `banned_expressions.md`。

## 局限

说几个这套东西做不到的。

模型本身的英文写作能力就是天花板。英文底子好的同学可能自己写比系统出的更好。行业内部信息（某个项目今年偏好变了、某个教授不怎么回套磁邮件）AI完全拿不到。素材挖掘的效果也严重取决于学生自己愿意投入多少时间在对话里，碰上每个问题都回答"还行"的人，系统也变不出东西来。

## 迭代

第一版是基础架构。拿真实case跑了几轮之后发现一堆问题（中英文风格怎么迁移、对话中断了怎么续接、推荐信和PS内容撞了怎么办），改了一轮。然后我们自己做对抗测试，故意拿各种极端场景去破——学生撒谎怎么办、两个Agent意见打架怎么办、上下文爆了怎么办——又改了一轮。最后用一个虚构的申请人端到端跑完全流程，把衔接上的缝隙补了。测试过程记在 `walkthrough_test.md` 里。
