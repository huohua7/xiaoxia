# 🦐 小虾 XiaoXia — AI多Agent研究生文书生成系统

> 6年留学文书实战经验，4位不同学科背景的资深顾问，把脑子里的东西掏出来教给了AI。

## 这个项目怎么来的

我们是一个做研究生申请文书的小团队，从2019年干到现在，经手过的案例覆盖了美国、英国、香港、新加坡的硕士和博士申请，累计帮超过400个学生拿到过offer。

团队四个人，背景差得很远。做社科方向（教育学、公共政策、传播学）的那位带过最多跨专业申请的case，最擅长帮"不知道自己有什么好写的"学生找到叙事角度，对社科类项目看重的"批判性思维""社会关怀"叙事套路和反套路都很熟。工科方向（CS、EE、ME）的是技术背景出身，自己就是理工科硕士毕业，最清楚工科文书里"项目经历怎么写才不像在抄简历"这个老大难问题。理科方向（化学、生物、统计）的做过最多PhD申请，对Research Statement和套磁信最有心得，他一直说理科教授招学生时在文书里真正看的就一件事："这个人对我的研究方向有没有真的了解"。商科方向（金融、MBA、管理学）的经手过不少MBA和商科硕士的essay，对"Why this program"和"Career Goal"类问题的回答策略最有经验，也是团队里最关注行文风格和可读性的人。

我们几个人平时做文书有一堆不成文的规矩，比如"开头绝对不准用名言引用""项目契合段至少要提一个具体教授的名字""推荐信覆盖了研究能力的话PS就少写研究多写动机"。这些东西散落在各自的脑袋里、微信群的聊天记录里、给新人培训的语音条里，从来没有系统整理过。

2025年初我们开始认真思考一件事：我们这些年积累下来的判断力，到底能不能被系统化地表达出来。不是让AI替我们写文书，而是把我们做咨询时的思考过程拆解成一套可执行的规则，让AI按照我们的方法论去工作。

这件事花了比我们预计久得多的时间。一个好的文书顾问在工作中做的大量决策是隐性的，比如你听学生讲完一段经历之后，直觉告诉你"这个点值得挖"或者"这段不能用"，你很难说清楚这个直觉背后的判断标准是什么。我们花了几个月的时间互相拷问，把彼此脑子里那些模糊的经验逼成了明确的条件和规则，到底是什么信号让你觉得一段经历值得深挖，你判断"该换角度了"的时候脑子里其实在算什么。

最终搭出来的是一套多Agent协作系统。每个Agent对应我们团队中一个具体角色的工作方式：interviewer 的提问节奏和追问策略来自我们做学生访谈时验证过的方法，strategist 的叙事架构设计来自我们内部讨论选题时的决策框架，reviewer 的审核维度来自我们互相改稿时积累的质量标准。

我们这行说到底是个信息差生意。好的文书顾问贵，是因为他脑子里有判断力，而判断力来自经验的积累。但我们一直觉得这事有点不对劲：一个家庭条件好的学生可以花几万块买到专业指导，一个同样优秀但预算有限的学生就只能自己硬写，最后产出的文书质量差距和他们本人的能力差距完全不成比例。

现在AI让这件事有了另一种可能。我们把经验灌进去，开源出来，预算有限的同学至少能拿到一套及格线以上的专业指引。它替代不了真人顾问对你个人情况的细致把握，但它能帮你避开大部分常见的坑，把有限的精力花在真正重要的地方。

## 这个系统能做什么

把研究生文书申请的完整流程拆成7个专业Agent的协作。你把配置文件喂给Claude（通过Claude Code、Project Knowledge、或System Prompt），它就能像一个有经验的留学顾问团队一样工作。

完整流程覆盖从竞争力诊断、素材挖掘对话、叙事策略规划，到英文文书撰写、质量审核、跨文书一致性检查、面试准备。支持 Personal Statement、Statement of Purpose、Diversity Statement、Research Statement，以及100到300字的短问答 supplemental essay。

## 系统架构

```
┌─────────────────────────────────────────────────┐
│                  Coordinator                     │
│            （中央协调器，管理全流程）                │
└──────┬──────┬──────┬──────┬──────┬──────┬───────┘
       │      │      │      │      │      │
       ▼      ▼      ▼      ▼      ▼      ▼
  Interviewer  Program    Strategist  Writer   Reviewer +        Interview
               Researcher                     Consistency       Coach
                                               Checker
  素材挖掘     项目调研     策略规划   文书撰写  质量审核+一致性    面试准备
```

工作流分六个阶段走。阶段零由 Coordinator 和 Program Researcher 并行完成竞争力诊断和目标项目分析。阶段一是 Interviewer 和用户进行三阶段对话（破冰、深挖、模式发现），产出理解文档。阶段二由 Strategist 基于理解文档和项目分析设计核心叙事、构建故事模块、做跨项目素材分配。阶段三 Writer 按策略简报撰写英文文书。阶段四是用户审阅、Reviewer 六维度审核、Consistency Checker 跨文书一致性检查三方并行。阶段五由 Interview Coach 生成模拟面试问题和风险评估。

## 我们塞进去的关键经验

这些是做了几百个case之后总结出来的，也是这套系统和"直接让AI写文书"最大的区别。

文书的价值80%在写作之前。大部分学生找人帮忙写文书，以为核心工作是"写"。其实核心工作是写之前的素材挖掘和策略规划。同样的经历，找到正确的叙事角度和找不到，出来的文书说服力差十倍。所以系统的前三个阶段（诊断、素材挖掘、策略规划）才是重点，写作只是最后的执行。

小白申请人需要"脚手架式提问"。问"谈谈你的实习经历"，90%的学生会卡住或者给你一段简历式的描述。问"你在实习的时候每天到公司第一件事做什么"，他们就能开始说了。Interviewer 的整套提问策略都是围绕这个逻辑设计的。

跨经历模式是文书的灵魂。好的PS有一条贯穿全文的主线，这条主线来自申请人不同经历之间的隐含关联。我们管这个叫"连点成线"，系统的 Interviewer 在第三阶段专门做这件事。

推荐信和PS不应该说同一件事。很多学生的推荐信和PS高度重复，都在说"研究能力强"，浪费了推荐信这个独立信息通道。系统会在素材挖掘阶段收集推荐信信息，策略阶段基于推荐信已经覆盖的维度来调整PS的侧重，让两者形成互补。

冲刺校和保底校的文书其实应该有不同的写法。冲刺校应该激进一些，最大化你的独特卖点、不提弱点；保底校可以自信一些，多讲"我能为项目贡献什么"。系统会基于诊断阶段的分层结果给每篇文书标注不同的写作基调。

最后一个也是我们花了最长时间才想明白的：AI痕迹的真正问题在结构层。禁掉"ignited my passion""transformative experience"只是表面功夫。真正让一篇文书读起来"像AI写的"的是每段长度差不多、每个论点恰好配一个例子、全文情感从头到尾平滑上升没有任何起伏。系统的 Reviewer 做的是词汇和结构双层检测，结构层的检测权重更高。

## 快速开始

**用 Claude Code（推荐）：** 克隆仓库后，把 `CLAUDE_CODE_PROMPT.md` 的内容放到项目的 CLAUDE.md 里，开始对话，系统自动引导全流程。

**用 Claude Project Knowledge：** 在 claude.ai 创建新 Project，将 `CLAUDE.md`、所有 `agents/*.md`、`banned_expressions.md` 上传到 Project Knowledge。在 Project Instructions 中写"你是一个研究生文书生成系统的协调器，请阅读 Project Knowledge 中的 CLAUDE.md 了解你的完整配置，根据 agents/ 目录下的文件了解每个Agent的定义，当用户提供申请材料时按照六阶段工作流开始工作。"然后开始对话。

**直接喂 System Prompt：** 上下文紧张的话按优先级塞：`CLAUDE.md` 核心部分最重要，然后是 `interviewer.md`，再然后是 `writer.md` 加 `banned_expressions.md`，其他Agent按需加入。

## 防护机制

系统内置了好几层防护，因为我们在实际业务中吃过亏。

最早加进去的是事实核验。AI写文书有个毛病，它特别喜欢"把故事讲圆"，素材里没有的细节它会自己编一个补上去，读起来很通顺但内容是假的。所以我们要求文书中每个叙事点必须追溯到用户在对话中确认过的事实，writer 找不到依据的内容直接拦截。

细节密度异常检测是后来加的。我们碰到过学生说自己"负责了核心系统重构"，追问两轮连用了什么技术栈都答不上来。这种情况系统会在理解文档里做标注，后续写文书的时候自动降低对这段素材的依赖，面试准备环节也会把它标成极高风险。

AI痕迹这块我们最初只做了个禁用词清单，后来发现不够。真正让招生官觉得"这文书不对劲"的是结构上的机械感：每段差不多长、每个论点刚好配一个例子、情感从头到尾稳步上升。所以 Reviewer 现在做两层检测，结构层的权重比词汇层高。

还有一条硬规则是我们反复犹豫之后加的：如果对话结束后能用的素材连一段完整经历都凑不出来，系统直接拒绝往下走，给用户几个问题让他们回去想清楚再来。素材不够的时候硬写出来的文书一定是空的，交给学生是害人。

## 局限性

这套系统最大的天花板就是模型本身。Claude的英文写作能力决定了文书质量的上限，英文底子好的申请人自己写可能比系统产出更好。

另一个绕不过去的问题是信息差。"这个项目今年换了招生主任偏好变了""这个教授其实不怎么招国际生"——这些只有在行业里泡着的人才知道，AI没有渠道获取，系统能做的只是基于公开信息做推断。

素材挖掘的效果完全取决于用户愿意投入多少。碰上每个问题都回答"还行"的用户，系统也没法凭空变出故事来。Program Researcher 的项目调研也一样，有搜索能力的环境下表现还行，没有的话就只能靠训练数据里的旧信息做粗略推断。

## 适配其他场景

底层的多Agent协作框架不局限于留学文书。我们自己内部试过把 Program Researcher 的角色替换成公司调研来做求职文书，流程跑得通。理论上任何需要"先深入了解一个人，再基于了解做策略，最后执行产出"的创作任务都能套这个框架。核心可复用的东西是理解文档机制、分层读取、种子内容加改写、多角色并行审核这几个设计模式。

## 迭代历史

这套系统不是一次性写出来的。最初的版本只有基础的Agent分工和工作流，拿去跑了几个case之后发现一堆实操问题：中英文风格怎么迁移、用户中途离开了怎么续接、推荐信和PS的内容撞了怎么办。改完这些之后我们又自己做了一轮对抗测试，故意模拟各种极端情况去破坏系统，用户撒谎怎么办、两个Agent意见打架怎么办、上下文爆了怎么办，逐个设计了防护。最后用一个虚构的申请人端到端跑了一遍完整流程，又修了七个衔接上的缝隙。`walkthrough_test.md` 里有第四轮测试的完整记录。
